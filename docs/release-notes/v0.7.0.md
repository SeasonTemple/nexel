# v0.7.0 — absorb netops kernel hardening + plugin install support

Release date: 2026-05-25

## English

### Added

- **State health helpers** bound to every read-state command path. The
  new `assertValidExistingState(state)` converts structural
  `state.json` issues into `CommandError(ERR_STATE_INVALID)` with the
  full findings list; `assertStateSelectionsKnown(state, manifest)`
  accumulates every selection id that the current manifest no longer
  knows and throws `CommandError(ERR_NOT_INSTALLED)` with the cleanup
  list. Install / export / doctor bind both; uninstall / update /
  repair bind only the structural check so recovery paths stay
  operable. (Plan U1)
- **Runtime dependency engines.node floor check** in `release-preflight`.
  For every runtime dependency in `package.json`, the new
  `runtime-deps-engines-compatible` check parses
  `engines.node` from `package-lock.json` and verifies the lowest
  acceptable version is `<=` the host's `engines.node` floor.
  Disjunctive ranges (`^22.22.2 || ^24.15.0 || >=26.0.0`) handled.
  Exports the `rangeMinimum` helper for downstream reuse. (Plan U2)
- **ProductConfig plugin metadata fields** —
  `repositoryUrl`, `pluginName`, `marketplaceName`, `pluginAuthor`,
  `pluginDescription`. All optional; `pluginName` derives from
  `productName`, `marketplaceName` derives from
  `${productName}-marketplace`. `REQUIRED_FIELDS` stays at the SPI v1
  frozen five — pure-installer downstreams have a zero-burden upgrade
  path. (Plan U3)
- **SPI v1.2** — `pluginInstallInstructions` signature widened from
  `() => string` to `(productConfig) => string`. The three in-repo
  adapters (Claude / Codex / OpenCode) interpolate
  `productName`, `pluginName`, `marketplaceName`, and `repositoryUrl`
  into the rendered text. Lazy-validates missing `repositoryUrl` with
  an actionable error string. See ADR-0010. (Plan U4)
- **Sample-product three-platform plugin layout.** Adds
  `.claude-plugin/{plugin,marketplace}.json`,
  `.codex-plugin/plugin.json`,
  `.agents/plugins/marketplace.json`, and `.opencode/INSTALL.md` to
  `examples/sample-product/` so downstream products have a complete
  reference to copy. `package.json.exports` adds a subpath for the
  OpenCode plugin entry; `package.json.main` deliberately stays at the
  kernel barrel. See ADR-0011. (Plan U5)
- **`verifyMarketplaceLockstep` kernel helper** for cross-manifest
  version-field agreement. Spec shape:
  `{ path, pluginName?, expectedVersion? }`. Handles plugin.json,
  marketplace.json with pluginName selector, baseline-from-first
  semantics, and missing-file / malformed-JSON errors. Sample-product
  uses it as a contract test against `package.json`. Public API
  via `scripts/installer/index.mjs`. (Plan U6)
- **`scaffold` verb** (opt-in via
  `createCli({ enablePluginScaffolder: true })`). Generates the six-
  file plugin layout from a ProductConfig. Default-on dep audit warns
  when the downstream `package.json` omits `nexel` from
  `dependencies` / `peerDependencies`; `--no-deps` suppresses.
  Pure-installer downstreams never see the verb in `--help`. Exports
  `scaffoldPlugin` (pure function) and `dispatchScaffold` (argv
  adapter) for advanced callers. (Plan U7)
- Two ADRs: ADR-0010 (SPI v1.2) and ADR-0011 (sample-product plugin
  layout as v0.7 reference).

### Changed (breaking)

- **`package.json.engines.node`** bumped from `>=18` to `>=22.22.2`.
  Forced by U2's audit: `write-file-atomic@8.0.0` requires Node
  22.22.2 / 24.15.0 / 26+; `@clack/prompts@1.3.0` requires Node
  20.12+. The previous `>=18` floor advertised compatibility nexel
  did not actually have. **npm install on Node < 22.22.2 will now
  refuse**; CI environments should ensure runners are on a supported
  major.

### Notes

- `SPI_DEFAULTS` reference identity and per-field `Function.length`
  are documented as out-of-contract surface as of v0.7. Adapter
  authors must not write detection code against either.
- Plugin install support is a complete reference, not a contract:
  the on-disk layout under `examples/sample-product/` is stable
  through the v0.7 window; major-version reshapes will include
  migration guidance.

## 中文

### 新增

- **state 健康双校验**绑定到所有读 state.json 的命令路径。新
  `assertValidExistingState(state)` 把 `state.json` 结构问题转为
  `CommandError(ERR_STATE_INVALID)`，携带完整 findings；
  `assertStateSelectionsKnown(state, manifest)` 累计 manifest 已不
  认识的所有 selectionId 并抛 `CommandError(ERR_NOT_INSTALLED)`，
  消息带清理建议。install / export / doctor 同时绑定两者；
  uninstall / update / repair 仅绑定结构校验，保留 recovery 路径
  可用性。（Plan U1）
- **`release-preflight` 加入 runtime dependency `engines.node` floor
  校验**。新 `runtime-deps-engines-compatible` 检查从
  `package-lock.json` 解析每个 runtime dep 的 `engines.node`，断言
  最低可接受版本 `<=` 宿主 `engines.node` floor。支持
  `^22.22.2 || ^24.15.0 || >=26.0.0` 形态 disjunctive ranges。
  公开 `rangeMinimum` 供下游复用。（Plan U2）
- **ProductConfig 增 plugin 元数据字段** —
  `repositoryUrl`、`pluginName`、`marketplaceName`、`pluginAuthor`、
  `pluginDescription`。全部可选；`pluginName` 派生自 `productName`、
  `marketplaceName` 派生自 `${productName}-marketplace`。
  `REQUIRED_FIELDS` 保持 SPI v1 冻结五字段——纯 installer 下游零
  迁移代价。（Plan U3）
- **SPI v1.2** — `pluginInstallInstructions` 签名由 `() => string`
  扩为 `(productConfig) => string`。三个内置 adapter
  （Claude / Codex / OpenCode）内插 `productName`、`pluginName`、
  `marketplaceName`、`repositoryUrl`。缺 `repositoryUrl` 时
  lazy-validate 返回可操作错误文本。详见 ADR-0010。（Plan U4）
- **sample-product 三平台 plugin layout 补全**。在
  `examples/sample-product/` 新增
  `.claude-plugin/{plugin,marketplace}.json`、
  `.codex-plugin/plugin.json`、
  `.agents/plugins/marketplace.json`、`.opencode/INSTALL.md`，下游
  可直接抄。`package.json.exports` 新增 OpenCode plugin entry 子
  路径；`package.json.main` 故意保持指向 kernel barrel。
  详见 ADR-0011。（Plan U5）
- **`verifyMarketplaceLockstep` kernel helper** 跨 manifest 校验
  `version` 字段一致。spec 形态：
  `{ path, pluginName?, expectedVersion? }`。支持 plugin.json、
  marketplace.json + pluginName 选择器、baseline-from-first 语义，
  以及缺文件 / 坏 JSON 的错误抛出。sample-product 用其作 contract
  test 对照 `package.json`。从 `scripts/installer/index.mjs` 导出。
  （Plan U6）
- **`scaffold` verb**（opt-in，通过
  `createCli({ enablePluginScaffolder: true })` 开启）。从
  ProductConfig 生成六文件 plugin layout。默认开 dep audit，下游
  `package.json` 缺 `nexel` dependencies / peerDependencies 时发出
  warning；`--no-deps` 关闭。纯 installer 下游 `--help` 中看不到该
  verb。同时导出 `scaffoldPlugin`（纯函数）与 `dispatchScaffold`
  （argv 适配器）供高级使用。（Plan U7）
- 两份 ADR：ADR-0010（SPI v1.2）与 ADR-0011（sample-product plugin
  layout 作为 v0.7 reference）。

### 变更（破坏性）

- **`package.json.engines.node`** 由 `>=18` 提升到 `>=22.22.2`。
  U2 的 audit 强制：`write-file-atomic@8.0.0` 要求 Node 22.22.2 /
  24.15.0 / 26+；`@clack/prompts@1.3.0` 要求 Node 20.12+。之前的
  `>=18` floor 宣告了 nexel 实际不具备的兼容性。**Node < 22.22.2
  上 npm install 将被拒绝**；CI 环境需确保 runner 在受支持的
  major 版本上。

### 说明

- `SPI_DEFAULTS` 引用身份与各字段 `Function.length` 自 v0.7 起文档
  化为 contract 外表面，adapter 作者不得据此判定 "未覆盖"。
- Plugin 安装支持是一份完整的 reference，不是 contract：
  `examples/sample-product/` 下的 on-disk layout 在 v0.7 窗口内稳
  定；后续 major 版本的形态调整会附迁移指南。
